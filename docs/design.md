# 設計

> 実際のMoodle認証画面で、Outlook Webからのコード取得、Moodle入力欄への設定、専用タブの終了まで確認済みである。Outlook Webの画面変更時には、検出方法を見直す。

## 構成

Chrome拡張機能だけで動かす。外部サーバー、独自バックエンド、メール転送は使わない。

## 構成要素

- Moodle用コンテンツスクリプト
  - 対象の二段階認証画面を検出する。
  - 初回は、Outlook Webの受信トレイ一覧を一時確認する処理への同意を表示する。
  - 6桁コードを入力欄へ設定する。
- Outlook Web用コンテンツスクリプト
  - メッセージ一覧から、`noreply@lms.rd.dendai.ac.jp` と `あなたの認証コードです。` が一致する対象メールを確認する。
  - 最初に固定送信元を確認し、一致した行だけ件名、受信日時、表示テキストを処理する。
  - Moodleの認証画面を最初に検出した時刻以降に届いたメールだけを候補にし、初回同意やOutlook表示の遅れで対象メールを取りこぼさない。
  - 条件一致した6桁コードだけを拡張機能内部へ渡す。
  - 受信トレイ一覧の準備後に10秒タイマーを開始し、停止メッセージ、取得成功、時間切れのいずれかで監視を終了する。
- バックグラウンド処理
  - 同意済みのMoodle認証画面を起点に、Outlook Webの専用タブを毎回前面で新規作成する。
  - 対象メールの監視開始・停止と、各スクリプト間の受け渡しを管理する。
  - 動作中だけタブID、認証開始時刻、監視開始時刻を`chrome.storage.session`に保持し、取得後または時間切れで削除する。
  - 同意状態だけを`chrome.storage.local`に保持し、オプション画面で取り消せるようにする。

## データの流れ

1. Moodle用コンテンツスクリプトが認証画面を検出した時刻を記録し、初回は利用者の同意を取得する。
2. バックグラウンド処理が同意状態を確認し、Outlook Webの専用タブを前面で開く。
3. Outlook Web用コンテンツスクリプトが受信トレイ一覧の表示後、対象メールだけを最大10秒確認する。
4. 6桁コードだけをバックグラウンド処理経由でMoodle用コンテンツスクリプトへ渡す。
5. Moodle用コンテンツスクリプトがコード欄へ入力する。送信操作はMoodle側に任せる。
6. コード取得時はMoodleタブを前面に戻してから、拡張機能が作成したOutlook Webタブを閉じる。10秒の待機時間切れではMoodleタブを前面に戻し、Outlook Webタブを残す。

## 状態表示と失敗時の動作

- Moodle上に、Outlook Webを開いたこと、認証メールを待機中であること、時間切れ時のOutlook Web確認案内を表示する。
- Outlook Web側の検出スクリプトでエラーが起きた場合、Moodle上にエラーを表示する。
- コード取得後は、認証コード自体を画面表示せずにMoodle入力欄へ渡す。
- Moodle側が入力を受け付けた応答を返した場合だけOutlookタブを閉じる。入力できない場合はOutlookタブを残して手動確認を案内する。
- 同意前はOutlook Webを開かず、同意を取り消したときは監視とタイマーを停止してOutlookタブを残す。

## 初回ログイン時の案内

Outlook WebにChrome内でログイン済みなら、そのまま動作する。未ログインなら、利用者本人が専用タブでログインする。パスワード入力は自動化しない。

## 権限の方針

- MoodleとOutlook Webに限定してアクセスする。
- 全サイトへのアクセス権限は要求しない。
- Microsoft Graph、メールAPI、メール転送、外部サーバーは使わない。
- オプション画面は同意状態の確認・取り消しだけを提供する。
