# 判断記録

## 2026-07-20: 大学Microsoft Graph APIを採用しない

### 背景

Outlookの公式APIを使えば、画面構造に依存せず認証メールを取得できると考えた。

### 確認結果

大学アカウントでGraph Explorerの`/me/messages`を実行すると、メール読み取り権限が不足していた。`Mail.Read`の同意を試みると、大学テナントの管理者承認が必要と表示された。

### 判断

大学管理者の承認を前提にしないため、不採用とする。

### 影響

メール取得は公式APIではなく、Chrome上のOutlook Webを対象にする。画面変更時に追随が必要になる。

## 2026-07-20: Outlook Webを使うChrome拡張機能を採用する

### 比較した選択肢

- Outlookデスクトップアプリの画面操作
- Gmailへのメール転送とGmail連携
- Thunderbird連携
- Outlook Webを使うChrome拡張機能

### 判断

Outlook Webを使うChrome拡張機能を採用する。

### 理由

- 大学管理者の承認を必要としない。
- メールを外部サービスへ転送しない。
- Windows専用のデスクトップアプリを配布するより導入が容易。
- Chrome Web Storeを通じて友達へ配布できる。

### 影響

利用者は初回だけChrome内でOutlook Webへログインする必要がある。

## 2026-07-21: Outlook Webの専用タブは前面で開く

### 背景

非アクティブな専用タブでは、Outlook Webの受信トレイが描画・更新されず、対象メールを取得できないことがあった。

### 判断

Moodleの認証画面を検出したら、Outlook Webの専用タブを前面で開く。

### 影響

認証中に画面がOutlookへ切り替わる。コード取得後は拡張機能が開いた専用タブを閉じる。受信トレイ表示後10秒以内に対象メールが見つからない場合は、Moodleへ戻って専用タブを残し、利用者が確認する。

## 2026-07-21: 初回同意後だけOutlook Webを自動確認する

### 背景

Chrome Web Store公開では、Outlook Webの受信トレイ一覧に表示される認証情報を一時処理することを、利用者へ明確に開示する必要がある。

### 判断

Moodleの二段階認証画面で、処理内容を説明して利用者が明示同意した場合だけ、Outlook Webを開いて自動確認する。同意状態は端末内に保存し、拡張機能の設定画面からいつでも取り消せるようにする。

### 影響

同意前・同意取り消し後はOutlook Webを開かず、認証コードを自動確認しない。取り消し時は実行中の監視とタイマーを止めるが、利用者が確認できるようOutlookタブは閉じない。

## 2026-07-23: 再審査では権限とデータ処理範囲をさらに限定する

### 背景

v1.0.0は、機能に不要な`tabs`権限を理由にChrome Web Storeで拒否された。追加監査により、`alarms`はストア版Chromeで10秒待機に使えず、静的コンテンツスクリプトと重複する`host_permissions`も不要であることを確認した。

### 判断

必須権限は同意状態と実行中セッションに必要な`storage`だけにする。Moodleは二段階認証パス、Outlook Webは`/mail/`配下だけに静的コンテンツスクリプトを限定し、Outlook側の10秒タイマーで待機終了を管理する。

### 影響

同意撤回やセッション中断では停止メッセージでOutlook監視を即時終了する。対象メールの固定送信元が一致した行だけを詳しく処理し、Moodleへの入力成功を確認できた場合だけOutlookタブを閉じる。
