# 調査・検証記録

## Moodleの二段階認証

- RD講義資料ページでは、ID・パスワード入力後にメールによる二段階認証画面が表示される。
- 認証メールには6桁コードが含まれ、コードの有効期限は10分と表示される。
- 対象メールはMoodleの専用送信元と件名で識別できる見込みがある。

## Microsoft Graphの検証

- 大学アカウントでGraph Explorerへログインできた。
- プロフィール取得は成功した。
- メール一覧取得は権限不足で拒否された。
- `Mail.Read`の同意には大学テナント管理者の承認が必要だった。

## Outlook Webの検証

- Chromeで大学Outlook Webへログイン済みの状態を確認した。
- 受信トレイのメッセージ一覧では、対象認証メールを次の3条件で判別できた。
  - 実際の送信元アドレス: `noreply@lms.rd.dendai.ac.jp`
  - 固定件名: `あなたの認証コードです。`
  - 受信日時: メッセージ行に完全な日時として表示される。
- 既存の対象メール4件と、Moodleで新たに発行した認証メール1件で同じ条件が一致した。
- 新着メールは受信トレイの先頭に表示され、一覧は日付順で並べ替えられていた。
- 判別にメール本文を開く必要はなかった。
- Outlook Webの一覧にある受信日時は分単位の表示である。認証開始時刻を分単位に切り捨て、その時刻以降を候補にすることで、同じ分に届いたメールの取りこぼしを避ける。

### 実装時の検出方針

- Outlook Webのメッセージ一覧にある行を対象にする。
- 送信元アドレスと件名が一致し、Moodleの認証画面を検出した後に受信したメールだけを候補にする。
- 受信日時を使って過去の認証メールを除外する。
- DOMのクラス名や会話IDには依存しない。送信元アドレス、件名、受信日時の表示要素を使う。

## 結合検証（2026-07-21）

- Outlook Webの実際のURLは`outlook.office.com`から`outlook.cloud.microsoft`へ遷移したため、両方を拡張機能の対象ホストに含めた。
- 専用タブを非アクティブで開くと受信トレイの描画・更新が遅れることがあった。前面で開く方式に変更すると、対象メールの取得を確認できた。
- 実際のMoodle認証画面で、認証画面の検出、Outlookからのコード取得、Moodle入力欄への設定、専用タブの終了まで確認した。
- 認証コード、メール本文、認証用リンク、セッション情報は記録しない。

## 注意

- 認証コード、認証用リンク、セッション情報、メール本文をスクリーンショット・文書・Gitへ保存しない。
- 認証コードは必要な瞬間だけ拡張機能内で扱い、保存しない。
